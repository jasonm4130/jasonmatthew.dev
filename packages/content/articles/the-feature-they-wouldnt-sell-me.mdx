My wife lifts. She was running a Starting Strength novice program through the official app, and she hit the wall every novice hits: linear progression stops working. The weight you added last session is the weight you fail this session. The textbook answer is to stop adding load every workout and switch to something that reacts to how the session actually went.

The app would not let her do that. It is built around the linear model and holds you there. The one thing she needed, the ability to leave linear progression when her body was done with it, was the one thing the product had decided not to give her.

That is the part worth sitting with. The gap was not technical. Reacting to a set's reps and adjusting next week's weight is arithmetic. The gap was a product decision, made by someone else, that did not account for her. A year ago that is where it would have ended: live with the app, or pay for a different one and hope it fits better.

It did not end there. I built her the feature instead. The app is called Ledger, it runs at [ledger.jasonmatthew.dev](https://ledger.jasonmatthew.dev), and the source is [on GitHub](https://github.com/jasonm4130/ss-webapp). This is what it does, the two pieces I am actually proud of, and the honest story of how long it took.

## The Autoregulation They Withhold

The feature she wanted is a weekly AMRAP progression. You do your work sets, the last one is "as many reps as possible," and the reps you hit decide next week's load. No fixed jump every session. The set tells the program what to do.

The whole engine is one small pure function with a three-zone hold band:

```ts
/**
 *  reps < floor            -> decrease (-2.5, or -5 when well short)
 *  floor <= reps < ceiling -> hold (you hit your sets, weight stays)
 *  reps >= ceiling         -> increase, larger the further over (+2.5 / +5 / +7.5)
 */
export function topSetDeltaKg(amrapReps: number, floor: number, ceiling: number): number {
  if (amrapReps < floor) return amrapReps - floor <= -3 ? -5 : -2.5;
  if (amrapReps < ceiling) return 0;
  const over = amrapReps - ceiling;
  if (over >= 4) return 7.5;
  if (over >= 2) return 5;
  return 2.5;
}
```

That is the thing the paid app would not do. Hit the top of the band and the weight climbs, more when you crush it. Land in the middle and it holds you there until you earn the next jump. Fall short twice and it deloads. The program follows the lifter instead of the calendar. A repeated short session also trips a stall counter that shrinks the top set, so a bad week does not compound.

It is not a clever idea. It is a known training method. The point is that it was withheld, and withheld things used to mean you were stuck.

## Why Adding It Was a One-Line Change

Here is the part that made this a side project and not a rewrite. Every lift in Ledger carries a `method`, and a method is a small self-contained module behind a single interface:

```ts
interface MethodModule<S> {
  kind: MethodKind;
  init(lift: Lift): S;
  prescribe(state: S, lift: Lift): Prescription;
  advance(state: S, completed: SetResult[], ctx: AdvanceCtx, lift: Lift): { newState: S; event?: ProgressionEvent };
}
```

`prescribe` decides what to put on the bar. `advance` reads the set you logged and returns the next state. Both are pure functions, no database, no React. Methods are looked up by `kind` through a registry:

```ts
const REGISTRY = {
  linear: linearModule,
  manual: manualModule,
  weeklyAmrap: weeklyAmrapModule,
};
```

Adding the autoregulation she needed was one new module, one arm on a union type, and one line in that registry. No database migration. The progression state lives inside `lift.method.state` as an opaque blob, so a new method ships its own shape without touching the schema. There is even a regression test that fails if any method writes back to the old top-level fields, because the cost of a clean boundary is that you have to defend it.

Three methods ship today: linear, manual log-only, and the weekly AMRAP. The structure means the fourth is an afternoon, not a project.

## The Timer That Survives the Tab Closing

The piece I find most interesting is the rest timer, because the obvious way to build it is wrong.

A rest timer is a countdown. The naive version is a `setTimeout` in the page that fires a notification when it hits zero. On a phone that fails exactly when you need it: you start your rest, lock the screen or switch apps, the browser throttles or kills the timer, and the notification never comes. The clock cannot live in a tab the user is about to walk away from.

So it does not. When you start a rest, the app posts your push subscription and the duration to a Worker, which hands them to a Durable Object keyed by your subscription endpoint:

```ts
export class RestTimerDO extends DurableObject<Env> {
  async schedule(subscription: PushSubscriptionJSON, seconds: number) {
    await this.ctx.storage.put('subscription', subscription);
    await this.ctx.storage.setAlarm(Date.now() + Math.max(1, seconds) * 1000);
  }

  async alarm() {
    const sub = await this.ctx.storage.get<PushSubscriptionJSON>('subscription');
    if (!sub) return;
    const status = await sendWebPush(sub, this.env);
    if (status === 404 || status === 410) await this.ctx.storage.delete('subscription');
  }
}
```

The clock is now server-side. The Durable Object alarm fires whether the tab is open, backgrounded, or closed. When it fires it sends a payload-less Web Push that wakes the service worker, which shows "Rest complete." Keying the object off the push endpoint means every device gets its own timer with no accounts and no shared state, and if the push endpoint is dead (a 404 or 410) the object drops it so it never retries a subscription that is gone.

There is no third-party push library in here. The VAPID authentication is hand-rolled against the spec with Web Crypto: build the ES256 JWT, sign it with the private key, send the tickle. About sixty lines. Doing it by hand was the part I most wanted to understand, and it is the kind of thing that is genuinely satisfying to get working on a free tier with no backend database anywhere in the system. Your training data lives in IndexedDB on your phone. The only server-side state is a timer that exists for the length of one rest.

## Mostly From My Phone, And How Long It Actually Took

I want to be honest about the build, because the honest version is the better story.

The first usable version came together over a lunch break, a lot of it typed into Claude Code on my phone. That part is true and it still surprises me: standing at a counter, describing the feature, watching working code come back. But "an afternoon" would be a lie, and the git history would catch me out. Ledger is a couple of evenings on top of that lunch. It has over a hundred tests, a dark theme, a setup wizard, kg and lb handling with plate-loading math, backup and restore, and the progression engine above. The polish is where most of the time went, and the polish is the point. Anyone can get a prototype in an hour now. The distance from prototype to "my wife uses this every session" is still real work, and AI compresses that distance without erasing it.

That is the actual democratisation, and it is narrower and more useful than the no-code utopia version. I did not avoid knowing things. I still had to know what autoregulation should do, why the timer belonged in a Durable Object, what a clean method boundary buys you. The knowledge was load-bearing. What collapsed was the distance between having the knowledge and shipping the thing. The feature a company decided not to sell my wife took me two evenings to give her, because the cost of going from "I know how this should work" to "it works on her phone" fell through the floor.

The next person's frustration with the next product does not have to end at the app's edge anymore. That is the part I keep thinking about. Not that software is free, but that the things you were quietly told you could not have are, increasingly, things you can just build.
